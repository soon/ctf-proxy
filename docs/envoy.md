# Envoy proxy configuration

The proxy is a **transparent** interceptor. iptables (`ctf_proxy/bin/iptables-config.py`)
REDIRECTs traffic destined for each configured service port to one of Envoy's listeners,
and the `original_dst` listener filter restores the original destination so Envoy can forward
to the real service. Per-service routing therefore lives in iptables — Envoy itself is a fixed
config generated from `ctf_proxy/proxy/envoy.template.yaml` (the `{INTERCEPTOR_FILENAME}`
placeholder is filled by `ctf_proxy/bin/refresh-envoy.sh`).

## Listeners

| Listener | Port | Service types (in `config.yml`) | Interceptor |
|----------|------|---------------------------------|-------------|
| `http_listener` | 15001 | `http`, `https`, `ws`, `wss` | HTTP wasm filter |
| `tcp_listener`  | 15002 | `tcp`, `udp`, …                | network (TCP) wasm filter |

A single interceptor wasm is built (`interceptor/Dockerfile.build`) and selects its mode at
load time from an env var passed in each filter's `vm_config.environment_variables`
(`CTF_PROXY_IS_HTTP` / `CTF_PROXY_IS_TCP`); it panics if neither is set.

## HTTPS interception (TLS termination + re-encryption)

`http_listener` handles **both** plaintext HTTP and HTTPS on the same port. The
`tls_inspector` listener filter detects TLS on the connection:

- **TLS connections** match the `transport_protocol: tls` filter chain, which terminates TLS,
  runs the HTTP interceptor on the decrypted traffic, then re-encrypts to the original
  destination via the `https_passthrough` cluster (`auto_sni` forwards the request Host as the
  upstream SNI; the backend certificate is not verified).
- **Plaintext connections** fall through to the `default_filter_chain` →  `http_passthrough`.

Because TLS is detected per-connection, a service that serves both HTTP and HTTPS (e.g. a
`:80` → `:443` redirect plus a TLS vhost) is handled correctly as long as both of its ports
are classified as `http`/`https` so iptables sends them to 15001.

### Single shared certificate

Termination uses one self-signed certificate for **all** services
(`ctf_proxy/proxy/tls/{cert,key}.pem`, generated by `setup.sh`). This works because in
attack/defense CTFs neither attackers (`curl -k`) nor, typically, the SLA checker validate the
service certificate — and the interceptor never needs a service's private key, since it opens a
fresh TLS connection to the backend rather than impersonating it cryptographically.

## Special case: per-service certificate pinning

If a client of a specific service **does** validate or pin that service's certificate (some SLA
checkers do), the single shared cert will fail its check. There is no automatic mechanism for
this — the certificate is chosen during the TLS handshake, before any L7/wasm logic runs, so it
cannot be selected by request content. It must be done by **manually editing the proxy config**.

Add a dedicated TLS filter chain to `http_listener` in `ctf_proxy/proxy/envoy.yaml` (the
generated file; mirror the change in `envoy.template.yaml` to survive a regen) that selects the
real certificate, and mount that service's `cert.pem`/`key.pem` into the Envoy container.

Selection happens on data available at handshake time only — the **original destination port**
(restored by `original_dst`), the **destination IP** (`prefix_ranges`), or the client **SNI**
(`server_names`). Destination port is usually the cleanest key, since services are enumerated
per-port:

```yaml
filter_chains:
# Pinned service on original destination port 8443 → present its real cert.
- filter_chain_match:
    transport_protocol: tls
    destination_port: 8443
  transport_socket:
    name: envoy.transport_sockets.tls
    typed_config:
      "@type": type.googleapis.com/envoy.extensions.transport_sockets.tls.v3.DownstreamTlsContext
      common_tls_context:
        tls_certificates:
        - certificate_chain: { filename: /etc/envoy/tls/service-8443.crt }
          private_key:       { filename: /etc/envoy/tls/service-8443.key }
  filters: [ … same http_connection_manager stack … ]

# Generic TLS chain (shared self-signed cert) for everything else.
- filter_chain_match:
    transport_protocol: tls
  transport_socket: { … DownstreamTlsContext with the shared cert … }
  filters: [ … ]
```

Envoy picks the most specific matching chain, so the generic `transport_protocol: tls` chain
remains the catch-all. Other selectors if port is not unique enough:

- **By destination IP:** `prefix_ranges: [{ address_prefix: 10.13.37.5, prefix_len: 32 }]`
- **By SNI:** `server_names: ["service.example"]` (the `tls_inspector` filter is already enabled).

The `filters:` block (HTTP connection manager + wasm + tap + router) must be duplicated in each
chain — only the `filter_chain_match` and `transport_socket` differ.
